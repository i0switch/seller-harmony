A. C04 / CAND-P0-01

分類: Stripe多重Checkoutによる二重課金・DB不整合
判定: 採用
深刻度: P0
再現可否: 条件付き再現
対象: stripe-checkout + stripe-webhook + memberships(DB)

概要

Webhook反映前に同一buyerが短時間でCheckoutを多重作成すると、Stripe側では複数subscriptionが有効化され得る一方、DB側は UNIQUE(buyer_id, plan_id) を前提とした upsert により 最新subscriptionで上書きされます。
その結果、古いsubscriptionがDBから追跡不能な孤児化状態となり、実質的な二重課金事故になります。

実害

多重課金

DB上は最新subscriptionのみ保持

古いsubscriptionの解約導線消失

カスタマーサポート介入が必要

対応内容 (C04)

4. 棄却した不具合
C01

判定: 棄却
理由:
「DB UNIQUEなしで重複処理が通る」という前提が誤りです。
stripe_event_id に UNIQUE 制約があるため、想定された二重処理はDB側で抑止されています。

評価

仕様である可能性: 高

弱い前提:

実DB制約未確認のまま、check-then-insertのみで重複通過すると断定していた

C05

判定: 棄却
理由:
is_active カラムを前提とした指摘でしたが、現行DBにはそのカラムが存在しません。
現行仕様では is_public + deleted_at による公開制御であり、主張は成立しません。

評価

仕様である可能性: 高

弱い前提:

存在しないカラムを基準に評価した

C08

判定: 棄却（過剰評価）
理由:
Step2でstate再検証しないのは事実ですが、JWT必須かつ前段でstate検証済みです。
したがって、直ちにCSRF脆弱性と断定できるものではなく、防御深度の議論に近いです。

評価

仕様である可能性: 中

弱い前提:

「CSRF成立条件」を過小評価していた

C10

判定: 棄却
理由:
ambiguous_server を明示返却して保護的に停止する実装です。
黙って誤サーバーへ付与する実装ではないため、脆弱性とは言えません。

評価

仕様である可能性: 高

弱い前提:

フォールバックと脆弱性を同一視した

6. 根本原因ベースの統合
6-1. Stripe整合性の問題群

対象: C02 + C03 + C04

根本原因

非同期イベント駆動での事前抑止不足

状態遷移の網羅不足

返金対象や更新対象を特定する解決キー不足

まとめ

Stripeまわりは個別バグというより、**「非同期イベントを前提とした整合性設計が弱い」**ことが根本です。

6-2. Discord OAuth契約不一致の問題群

対象: C06 + C07 + C08
加えて別レポートの CAND-P1-02 も同系統の重要論点です。

根本原因

2段階フローの契約が、実装/運用要件とズレている

redirectの扱い

error model

stateの責任境界

DB状態遷移とロール付与の分離

まとめ

Discord OAuth系は、「正常系の業務完了条件」が分割されすぎていることが問題です。

9. 監査結論
確認済みで重要な結論

最優先はStripe整合性修正です。

特に C04 は P0相当 で、二重課金事故に直結します。

次点で C02 / C03 は、課金成功後の利用不能や返金不整合を引き起こすため、リリース前修正が必要です。

Discord系では CAND-P1-02 が実害大で、オンボーディング完了条件の再設計が必要です。

ノイズとして扱えるもの

C01 / C08 / C10 は、現行ガードや成立条件の厳しさを踏まえると、優先度を上げすぎるべきではありません。

C05 は脆弱性ではなく、主にデッドコード・前提誤認の問題です。
