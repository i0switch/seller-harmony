# 現状分析とタスク管理レポート

プロジェクトの全体像および提供されている要件定義書（`要件定義_ai_optimized.md`）、現在のアーキテクチャスタックに基づき、システムを本番稼働させるために必要な情報をまとめました。

## 1. 未実装・未検証一覧 (Unimplemented / Unverified Features)
現状のアーキテクチャから、本番運用に向けて追加または深く検証が必要な主要実装項目です。

*   **Discord関連の厳格なエラー制御と競合回避**
    *   Discordのロール階層チェック（Botのロールが付与対象より上にあるかの事前検証）。
    *   同一サーバー内で複数プラン契約時のロール剥奪スキップ制御。
*   **Stripe Webhookの高度なハンドリング**
    *   `charge.dispute.created` / `closed` に対するチャージバック発生時のリスクフラグ(`risk_flag`)付与や即時ホールド機能。
    *   猶予期間（Grace Period）の経過と、Stripeのリトライ完了タイミングの完全な同期ロジック。
    *   `invoice.voided` など、請求無効時の確実なロール剥奪トリガー。
*   **テナント（販売者）向け運用ツール群**
    *   自テナントエンド会員の統合タイムライン機能UI（`stripe_webhook_events`とDBログの突き合わせ表示）。
    *   自動剥奪処理の停止・手動維持（オーバーライド機能）。
    *   権限付与エラー時の個別リトライ実行APIとUI。
*   **プラットフォーム管理者向け機能**
    *   プラットフォーム全体ベースのWebhook再試行キュー管理ビュー。
    *   通知配信機能（`system_announcements`）の管理画面。

## 2. 本番リスク (Production Risks)
稼働直後に直面しやすい技術的・ビジネス的リスクです。

1.  **Discord APIのRate Limit（制約）抵触**
    *   **リスク**: 大量の一斉サブスク請求失敗や更新時に、DiscordAPIへのRole付与/剥奪リクエストが集中し、Rate Limitによるパニックやキュー詰まりが発生する。
    *   **対策**: 処理のバッチ化、非同期ワーカーでの指数的バックオフつきリトライ（Rate Limitヘッダーの尊重）の確実な実装。
2.  **Stripe Connect決済の分配（Separate Charges and Transfers）の運用リスク**
    *   **リスク**: 返金発生時やチャージバック時にプラットフォームの残高が一時的にマイナスになるリスクや、複雑な分配ロジックのWebhook処理漏れ。
    *   **対策**: 返金やエラーのWebhookを確実に受け取り、ロールを確実に剥奪する処理。
3.  **Webhookの冪等性とDBのロック**
    *   **リスク**: Stripeは稀に同一イベントを複数回送信します。冪等性処理が不足していると、多重にDiscordAPIを叩くか、DBが意図しない状態遷移（ステートマシン破壊）を起こします。
    *   **対策**: `stripe_webhook_events` テーブルを利用した厳密な一意チェックと排他制御（トランザクション）の徹底。
4.  **不十分なCSRF対策（Discord OAuth）**
    *   **リスク**: 悪意あるユーザーが誘導を行い、本人の意図しないDiscordアカウントへ連携させてしまう。
    *   **対策**: OAuth2 `state` パラメータ生成と検証の徹底強化。

## 3. 優先修正・実装順 (Priority Fix Order)
リスク軽減とコア価値提供を確実にするための対応優先度です。

1.  **[最優先] 決済・権限付与のコアパイプライン安定化**
    *   Stripe Webhookの冪等性制御（Idempotency Keyチェックや、処理済みイベントのスキップ）の実装・見直し。
    *   Discord OAuthの `state` パラメータによるCSRF防御と意図しないアカウント紐付け防止機能のチェック。
2.  **[高] 状態遷移（ステートマシン）の厳格化・例外ハンドリング**
    *   `grace_period`（猶予期間）のロジック実装。Stripeのリトライ期間の整合性をテスト。
    *   Role競合保護（複数プラン契約時のRole誤剥奪防止）ロジック。
    *   チャージバック・返金時（`charge.refunded` 等）のアクション実装。
3.  **[中] テナント（販売者）向け保守機能（自己解決策の提供）**
    *   「権限復旧の手動リトライ」および「手動による一時オーバーライド機能」の実装（サポートコスト削減のため）。
    *   会員情報のタイムライン表示APIの完成。
4.  **[低] プラットフォーム管理者用ダッシュボード拡充**
    *   全体キュー管理、システム通知発信システムの実装。

## 4. 必要テスト項目 (Required Test Items)
Playwright E2Eを中心としたテストに追加・集中的に行うべき検証項目です。

*   **決済＆Role付与（正常系）**
    1.  Stripe Connectのテスト用アカウントでのオンボーディングフロー完遂。
    2.  エンド購入者によるCheckout成功 -> Discord OAuth連携によるGuild Join＆Role付与の即時反映。
*   **サブスクリプション状態遷移（異常系・エッジケース）**
    3.  **猶予期間の検証**: 意図的なカード決済失敗 -> ステータスが `grace_period` へ移行 -> 対象日数待機後にようやくRoleが剥奪されること。
    4.  **即時解約の検証**: 期間終了待たずに販売者からの強制即時キャンセル -> Roleの即時剥奪。
    5.  **期間満了解約の検証**: ユーザーの次回更新キャンセル (`cancel_at_period_end=true`) -> ステータスが `cancel_scheduled` となり、**有効期間満了日まではRoleが維持**され、期日に剥奪されること。
    6.  **Webhook重複テスト**: 同一イベントIDでWebhookを2連続で送信し、2回目の処理が行われずエラーも吐かないこと（冪等性の証明）。
*   **Discord・Stripeの制約テスト**
    7.  **Botロール階層負けエラー**: Botより上位のRoleを付与しようとした際の適切なエラーハンドリングと、販売者への警告フィードバック。
    8.  **ロール競合回避**: ユーザーがAプランとBプラン（両方ともRole"VIP"を付与）を購入。Aプラン解約時に"VIP"が剥奪されないことの確認。
*   **運用サポートテスト**
    9.  販売者ダッシュボードから特定メンバーへの強制連携リトライボタンが動作し、再試行ログが保存されること。
    10. 手動オーバーライド中（権限剥奪のブロック設定中）にStripeで解約されても、Roleが残ったままになること。
